# Ideation — how the design-harness plug-in should work given what we now know

## Why this exists

`docs/knowledge/lessons/2026-04-17-self-audit.md` confirmed the scaffolded skill passes its own quality rubric. But applying the skill to **5 real repos** (figma/sds + Bill's 4: plus-marketing-website, caricature, oracle-chat, lightning) exposed a blindspot the self-audit couldn't see: **our "harness" concept is too narrow**. We recognize artifacts we scaffold. We don't recognize harnesses other tools built, knowledge outside the repo, or MCP integrations that already bridge to external sources.

This doc consolidates every gap surfaced by real-repo recon into a concrete v1.1 improvement plan. Locks the list before implementation.

## Real-repo recon (the data)

Observed-vs-expected mode detection across 5 real repos:

| Repo | Owner | Surface | detect-mode said | Reality | Gap |
|---|---|---|---|---|---|
| figma/sds | Figma (OSS) | `.cursor/rules/`, 16 KB `.github/copilot-instructions.md`, `figma.config.json`, `.storybook/` | scattered (pri 4) | scattered + Figma-integrated + multi-IDE | P2 — mode OK but blind to Figma + multi-IDE signal |
| plus-marketing-website | Bill + 2 designers | CLAUDE.md, `.agent/` framework, `.cursor/mcp.json` (shadcn), `.github/workflows/sync-notion.yml`, `docs/setup-mcps.md`, Notion-integrated | scattered (pri 4) | **advanced + MCP-configured + Notion-sourced + Storybook-SSoT** | **P1** |
| caricature | Bill (solo) | `docs/plans/` with 6 plans following compound naming convention, `skill/`, `supabase/`, `todos/` | greenfield (pri 6) | **advanced — plan-driven workflow already running** | **P1** |
| oracle-chat | Bill (solo) | `.claude/launch.json`, 84 docs incl. extensive `docs/plans/` + `docs/PRD_*.md` | greenfield (pri 6) | **advanced — `.claude/` directory fully invisible to detect-mode** | **P1** |
| lightning | Bill (solo) | `.claude/skills/` (4 SKILL.md files), `.cursor/skills/`, 74 docs, `tokens/`, Chinese PDF user-handbook | greenfield (pri 6) | **advanced — repo literally ships its own Layer 2 skills** | **P1** |

**3 of 5 real repos get P1-wrong detection.** Running `/hd:setup` today would try to scaffold from scratch on top of existing harness machinery in 3 repos, and blindly mis-classify the other 2.

## Root cause

**Our harness-detection signals only catch what WE scaffold.** Missing signal families:

1. Harnesses built by other tools (`.claude/`, `.codex/`, `.agent/`, `.cursor/skills/`, compound-engineering plans in `docs/plans/`)
2. MCP configuration (`.mcp.json`, `.cursor/mcp.json`, `mcpServers` in any config)
3. External-source integration (Notion sync workflows, Figma config files, Google Drive links in docs, Linear integration)
4. Tool-of-truth signals (Storybook, design-token packages, style-dictionary configs, Figma tokens)

And because detection is shallow, the skill itself never thinks to ASK about these sources. So even when a designer sits there with a Notion workspace full of brand docs, the skill cheerfully scaffolds empty `docs/context/product/one-pager.md` waiting for them to type.

## Conceptual correction before the change list

**The skill is NOT a classifier that reads your repo and hands you a report.** It's a **layer-by-layer facilitator** that always walks L1 → L2 → L3 → L4 → L5 in order, and at each layer adapts to what it finds:

- If a layer already has material (other-tool harness, external tool, existing docs) → offer three choices:
  - **(a) Link** — point to the existing source from the hd-* structure (pointer file, not copy)
  - **(b) Critique** — apply relevant rubric, surface gaps, let user decide what to fix
  - **(c) Scaffold alongside** — leave the existing thing untouched; add the hd-* equivalent as a separate concern
- If a layer has nothing → seed questions + scaffold.
- **Never merge, never absorb, never overwrite** external harnesses.

Suggest `/hd:onboard` first only when the user hasn't already (heuristic: no prior `design-harnessing.local.md`). After that, always the five-layer walk.

## Twelve concrete v1.1 changes

Organized by the layer they affect, ranked by priority. P1 = ships in v1.1; P2 = ships in v1.2; P3 = post-v1.2.

### Detection layer (`scripts/detect-mode.sh` + SKILL.md Step 1)

**C1 (P1) — Detect other-tool harnesses.**
Add signals:
- `has_claude_dir` — `.claude/` exists
- `has_codex_dir` — `.codex/` exists
- `has_agent_dir` — `.agent/` exists (plus-uno convention)
- `has_external_skills` — `.claude/skills/` or `.cursor/skills/` exists with ≥ 1 SKILL.md inside
- `has_plans_convention` — `docs/plans/` exists with ≥ 3 files matching `YYYY-MM-DD-NNN-<type>-<slug>-plan.md`

Any of these → `mode: advanced` with sub-signal `other_tool_harness: true`. Skill must NOT attempt greenfield scaffolding.

**C2 (P1) — Detect MCP configuration.**
Scan and parse:
- `.mcp.json` (Claude Code)
- `.cursor/mcp.json` (Cursor)
- `.codex/mcp.json` (Codex)
- Any file matching `**/mcp.json`

Emit in JSON: `"mcp_configured": true, "mcp_servers": ["notion", "figma", "shadcn", ...]`. This is authoritative — we know exactly which servers are wired.

**C3 (P1) — Detect external-tool signals across the whole team ecosystem.**

Teams don't just use docs tools — they use analytics, design, diagramming, PM, and comms. Detection categories:

| Category | Signals (grep URLs + package deps + config files) |
|---|---|
| **Docs / wiki** | notion.com, notion.so, confluence.*, coda.io, obsidian.md, docs.google.com |
| **Design** | figma.com/file, figma.com/design, paper.design, pencil.dev, sketch.com |
| **Diagramming** | excalidraw.com, miro.com, whimsical.com, lucid.app, figjam |
| **Analytics / measurement** | amplitude.com, mixpanel.com, posthog.com, metabase, hotjar.com, fullstory |
| **PM / issue tracking** | linear.app, jira, github.com/.../issues, asana.com, monday.com |
| **Comms** | slack.com, discord.com, loom.com |
| **Sync workflows** | `.github/workflows/sync-*.yml` or `sync-*` in workflow names |

Emit per category: `{ category: [server, ...] }`. Surface ALL detected at Step 1.5 as a reminder — the user likely uses more tools than they'd spontaneously list, and many are useful context for different layers. Not every tool is harness-relevant for every team; the user tags which ones matter.

**C4 (P1) — Detect token-package / config source-of-truth signals.**
- `tokens/`, `style-dictionary.config.*`, `tokens.config.json` → `tokens_package_sot: true`
- `figma.config.json`, `figma-config.json` → `figma_config_present: true`

(Storybook-as-SSoT detection deferred to v2 per Bill — not essential near term.)

Used to seed Layer 1 (Context) and Layer 4 (Rubrics) with the right source-of-truth pointers.

### Workflow layer — facilitation before scaffolding

**C5 (P1) — External-tool discovery at the top of every flow.**

New reference file: `skills/hd-setup/references/external-tooling.md` (broader than "sources" — covers design, docs, analytics, diagramming, PM, comms). New Step 1.5, before the five-layer walk:

> "Before the layer walk, let me show you what I detected and ask what your team uses. Category by category.
>
> **Docs:** [list detected] — missing: which ones does your team use?
> **Design:** [list detected] — missing: which ones?
> **Diagramming:** [list detected] — ...
> **Analytics:** [list detected] — ...
> **PM/issues:** [list detected] — ...
> **Comms:** [list detected] — ...
>
> For each tool you name, we'll decide per-layer whether it's the source of truth, a pointer, or unused. For ones with MCP available ([list]), I can pull live content when it's helpful; for the rest, I'll record the pointer so your future self knows where to look."

Records answers in `design-harnessing.local.md` under new field `team_tooling:` (map of category → list of tools with per-tool role + MCP status).

**C6 (P1) — MCP-aware branching.**

If `mcp_configured: true` with relevant servers:

- **Notion MCP available** → offer to pull top-level pages, let user tag which are "product context" / "conventions" / "lessons", and write pointer files (not copies) to `docs/context/` / `docs/knowledge/`
- **Figma MCP available** → offer to pull design-system component names + token names into `docs/context/design-system/cheat-sheet.md`
- **Linear MCP available** → offer to scan recent issues for recurring design-decision patterns (seeds Layer 5 lessons)

If NOT configured but user says their docs live there → show the install command (same shape as plus-marketing-website's `docs/setup-mcps.md`) and offer to write a `setup-mcps.md` of our own.

**C7 (P1) — Unified five-layer walk workflow (replaces greenfield / scattered / advanced split).**

The skill has ONE primary workflow: `workflows/layer-walk.md`. It walks L1 → L5 in order. At each layer, it presents what it detected (existing docs, external-tool pointers, MCP-available sources) and offers three options:

- **(a) Link** — add a pointer file from hd-* structure to the existing content. Original untouched.
- **(b) Critique** — run the appropriate rubric over existing content; surface gaps; let user decide per gap what to do. Original untouched.
- **(c) Scaffold alongside** — seed + write an hd-* layer file, leaving the existing content untouched.

User can pick different options per layer. Default suggestion per layer is context-aware:
- Nothing detected at layer → scaffold
- Existing other-tool artifact → link (respects coexistence)
- Existing hd-* partial content → critique + fill gaps

**Never merge or overwrite any other-tool harness** (`.agent/`, `.claude/`, `.codex/`, etc.). If the user wants migration they can do it manually outside this skill.

The three per-mode workflow files from v1.0 (`greenfield.md`, `scattered.md`, `advanced.md`) become thin references loaded INTO `layer-walk.md` for their seed-question content, not standalone routes.

**C8 (P2) — Facilitation-mode seed questions.**

Every layer scaffold gets 1–3 seed questions before the file is written. Instead of writing `docs/context/product/one-pager.md` with `{{PRODUCT}}` placeholders, the skill asks:

- "Describe your product in one sentence for a new teammate."
- "Who's the user in one sentence?"
- "What's the single biggest design constraint you face?"

Then fills the template with their actual answers. Already partially present in greenfield.md Step 3 — extend to all workflows.

**C9 (P2) — Adaptive prompts based on team size.**

Reading `team_size` from user input:
- **Solo (1):** scratchpad-mode language. Skip graduation ceremony framing. "Your future self" instead of "your teammate."
- **Small (2–5):** "you and your co-designers" framing. Introduce graduation-readiness without pushing for formal RFCs.
- **Medium (6–20):** full five-layer recommendation incl. explicit graduation workflow.
- **Large (21+):** defer, suggest per-product harness (monorepo pattern, scenario T5).

### Template layer

**C10 (P2) — Optional solo-mode simpler templates.**

For `team_size == 1`, write shorter templates:
- `AGENTS.md`: skip the "Graduated rules" section (solo can graduate directly to AGENTS.md without the ceremony)
- `docs/knowledge/lessons/` template: compressed YAML frontmatter, shorter body

Only written when user confirms solo. Team templates stay the default.

**C11 (P3) — `design-harnessing.local.md` schema v2.**

Additive fields (backward-compatible):
- `external_sources: { notion: { root_page: "...", access: "api_key | mcp | none" }, figma: { file_url: "...", mcp: true }, ... }`
- `mcp_servers_at_setup: ["notion", "figma", ...]`
- `tool_of_truth: { design_system: "storybook | figma | tokens-package | ad-hoc", product: "notion | repo-only | elsewhere" }`
- `other_tool_harnesses_detected: [".claude/", ".agent/", "docs/plans/"]`

### Quality rubric layer

**C12 (P3) — New starter rubric: `external-source-hygiene.md`.**

For teams that rely on Notion/Figma/Linear, add a rubric for how well those links stay fresh:
- Broken-link detection (HEAD check on URLs in committed docs)
- Stale-link detection (Notion last-edited > 6 months old via MCP)
- Orphaned-source detection (file references a Figma URL that's no longer in the design system index)

Applied by `hd:review audit` when `external_sources` is non-empty in local.md.

## What this does for each of the 5 repos

Every flow is the same shape: onboard suggestion (if new) → Step 1.5 external-tool discovery → L1 walk → L2 walk → L3 walk → L4 walk → L5 walk. What differs is the **default option suggested at each layer** based on what's detected.

**figma/sds** — "You haven't run `/hd:onboard` yet — want a 5-min intro to the five-layer frame first? [yes/no/already know]. Either way, let's walk L1 through L5.
- **L1 (Context):** detected `.cursor/rules/usage-guidelines.mdc`, `.github/copilot-instructions.md` (16 KB), `figma.config.json`. Default suggestion: **link** (point from hd-* Context to these). Options: link / critique / scaffold alongside?
- **L2 (Skills):** nothing detected. Default: skip for now unless you want to author one.
- **L3 (Orchestration):** nothing detected. Default: skip.
- **L4 (Rubrics):** detected `figma.config.json`, token scripts in `scripts/` — design-system-compliance rubric may apply. Default: **scaffold** starter rubrics.
- **L5 (Knowledge):** nothing detected. Default: **scaffold** empty lessons dir so you have a place to capture going forward."

**plus-marketing-website** — "I see you ran harness-like setup before — your `.agent/` framework is detected. **I won't touch it.** Let's walk the hd-* layers and decide per-layer whether to link to `.agent/`, critique, or scaffold alongside.
- **L1:** detected `.agent/rules/100-project-context.md`, Notion sync workflow, shadcn MCP. Default: **link** `.agent/rules/` + **pull key Notion pages as pointers** (MCP available). Critique option available if you want review.
- **L2:** detected `.agent/skills/frontend-design/`. Default: **link**. You already have this layer — no need to duplicate.
- **L3:** nothing explicit. Skip or scaffold.
- **L4:** detected OKLCH token system + `src/stories/styles/`. Default: **scaffold** design-system-compliance rubric referencing `.agent/skills/frontend-design/references/tokens.md`.
- **L5:** nothing. Default: **scaffold** for go-forward capture."

**caricature** — "Solo repo, no prior harness. Walking L1–L5.
- **L1:** `docs/google-ai-studio-*.md` files detected (product context). Default: **link** those. Seed questions for the gaps (user, constraints).
- **L2:** `skill/` dir exists. Default: show contents, ask if it's hd-* material or something else; default **link**.
- **L3:** default **skip**.
- **L4:** Gemini prompts in `src/` — prompt-hygiene rubric could apply. Default: **scaffold**.
- **L5:** `docs/plans/` has 6 plans. Default: **critique** — surface patterns across plans that look graduation-worthy; scaffold `docs/knowledge/lessons/` from those patterns."

**oracle-chat** — "Solo repo, `.claude/launch.json` detected (will not touch). 84 docs.
- **L1:** PRDs in `docs/PRD_*.md` detected. Default: **link**. Seed questions for anything missing.
- **L2–L3:** skip unless you want.
- **L4:** chat / response quality is inherently rubric-able. Default: **scaffold** an output-quality rubric starter.
- **L5:** 84 docs incl. many plans. Default: **critique** + suggest ~5 lesson extractions from the most repeated patterns."

**lightning** — "Solo repo, `.claude/skills/` and `.cursor/skills/` both present (4 SKILLs each, will not touch).
- **L1:** Chinese PDF handbook + product docs detected. Default: **link** the PDF as a product-context pointer.
- **L2:** existing `.claude/skills/` — default: **critique** with the `skill-quality` rubric. I'll read each SKILL.md and produce a 9-section rating.
- **L3:** skip.
- **L4:** `tokens/` dir present. Default: **scaffold** design-system-compliance rubric.
- **L5:** `docs/plans/` extensive. Default: **critique** + seed lessons from patterns."

Notice the pattern: **the skill never says "I classified you as X"**. It shows what it detected, walks the layers in order, and offers informed choices per layer. Meet the user where they are — layer by layer.

## Ordering / effort estimate (with Claude coding)

Phase 1 — detection (C1–C4): **~40 min**
Phase 2 — workflow (C5–C7, incl. new `layer-walk.md`): **~60 min**
Phase 3 — facilitation polish (C8–C9 if we ship them in v1.1): **~30 min**
Phase 4 — re-run self-audit + smoke tests: **~15 min**
Phase 5 — pilot on figma/sds: **~30 min**
Phase 6 — walk through Bill's 4 repos: **~60 min total** (all four in one session)

**Total: ~4 hours.** Splittable anywhere. Phases 1–4 land in this plug-in repo as a PR; Phases 5–6 land as real harness improvements in the 5 target repos (separate branches Bill reviews + merges per repo).
